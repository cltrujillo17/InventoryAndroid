Professional Self-Assessment

Completing the Bachelor of Science in Computer Science at Southern New Hampshire University has been a process of progressive, cumulative learning. Early courses gave me foundational vocabulary and syntax. Later courses gave me design principles and engineering judgment. The capstone forced me to apply both simultaneously to a real artifact — an Android Inventory Management App — and evaluate it honestly against professional standards. That evaluation revealed something important: a program that compiled and ran without errors was nonetheless architecturally broken in ways that would have made it unmaintainable, insecure, and incorrect in production.

Collaborating in a Team Environment

Collaborative work in software engineering is about writing code that others can understand, review, and extend. CS 250 (Software Development Lifecycle) introduced me to Agile methodology; CS 320 (Software Testing and QA) reinforced that readable, testable code is a prerequisite for any collaborative environment. Throughout this capstone, every method includes Javadoc comments explaining its contract and the reasoning behind non-obvious decisions. The code review I produced for Milestone One was recorded for an audience of peers and a hypothetical manager — simulating the real industry practice of presenting work for group scrutiny.

Communicating With Stakeholders

The three milestone narratives were each addressed to a dual audience: a technical instructor needing engineering depth, and a general reader needing to understand why the choices mattered. Explaining why plain-text password storage is a vulnerability, what a rainbow table attack is, and why a per-user salt prevents it — in clear prose without assumed prior knowledge — required the same translation skill a software engineer uses when presenting a security proposal to a non-technical product team.

Data Structures and Algorithms

The Milestone Three enhancement demonstrated that algorithmic improvement is often about knowing which layer should perform which computation. SQL LIKE with an indexed column replaces in-memory list filtering. Validated ORDER BY constants replace Java-level sorting. A single O(n) low-stock scan reuses already-loaded data without an additional query. CS 260 (Data Structures and Algorithms) gave me the vocabulary to reason about these tradeoffs deliberately.

Software Engineering and Databases

The MVVM refactor in Milestone Two reflects principles from CS 250 and CS 360: separation of concerns, single responsibility, and an observable data layer. The Milestone Four database enhancement built on this: SHA-256 + salt password hashing, structured schema migrations, and background threading via ExecutorService — each reflecting professional-grade database engineering.

Security Mindset

CS 405 (Secure Coding) shifted my perspective: security is a property of design decisions made before a line of code is written. The SecureRandom salt generation, per-user salting, the decision to drop and recreate the users table during migration rather than leave old plain-text credentials in place, and parameterized SQL queries throughout — each reflects adversarial thinking applied at the design stage.

How the Artifacts Fit Together

All three enhancements were applied to a single artifact. This was deliberate. The MVVM architecture (Milestone Two) was the prerequisite for search and sort algorithms (Milestone Three): only once data flowed through a ViewModel could real-time filtering be wired to a TextWatcher cleanly. The database security improvements (Milestone Four) were meaningful only because Milestone Two had connected the UI to the database in the first place. A visitor reading the three enhancements in sequence traces a single engineering story: a prototype-quality app transformed into a production-quality one through deliberate, documented, and reasoned enhancements.

Enhancement One — Software Design & Engineering

MVVM Architecture LiveData Data Model Fix Security Patch

What Was Improved

  • Fixed the data model: Item.java gained a typed long id and int qty field — quantity was previously parsed from a plain string.
  • Introduced MVVM: InventoryViewModel now owns all data operations and exposes inventory as LiveData<List<Item>>; SecondFragment only observes.
  • Connected the database: Items finally persist across app restarts — the original app used an in-memory ArrayList that was never written to SQLite.
  • Removed bypass buttons: Two unauthenticated buttons that skipped the login screen were deleted from both the layout and Java code.
  • Upgraded GridAdapter: Extended ListAdapter with DiffUtil for efficient, animation-backed diffs; deletion is now ID-based, not position-based.
  • Fixed deprecated API: SmsManager.getDefault() replaced with getSystemService(SmsManager.class).

Course Outcomes Addressed

Outcome 3 (architectural tradeoffs — MVVM), Outcome 4 (LiveData, DiffUtil, Navigation Component), Outcome 5 (bypass removal, defensive validation).

Enhancement Two — Algorithms & Data Structures

SQL LIKE Search ORDER BY Sort Low-Stock O(n) Scan SQL Injection Prevention

What Was Improved

  • Real-time search: A TextWatcher fires a parameterized SQL LIKE '%query%' query on every keystroke. The idx_items_title index makes this sub-linear.
  • Sort options: A Spinner exposes four sort modes (newest, A-Z, qty low→high, qty high→low). Each maps to a validated ORDER BY constant — no Java-level sort is ever performed.
  • Low-stock detection: countLowStockItems() performs a single O(n) pass over the already-loaded list. A warning banner and red quantity labels give immediate visual feedback.
  • Security: Search uses parameterized queries; sort uses a whitelist of constants — no user input is ever concatenated into SQL strings.

Course Outcomes Addressed

Outcome 3 (algorithmic tradeoffs — DB-layer vs. in-memory), Outcome 4 (TextWatcher, Spinner, SQL), Outcome 5 (SQL injection prevention via parameterized queries).

Enhancement Three — Databases

SHA-256 + Salt SecureRandom Schema Migration Background Threading

What Was Improved

  • Password hashing: SecureRandom generates a 16-byte per-user salt. MessageDigest produces a SHA-256 hash. Only the hash and Base64-encoded salt are stored — never the plain-text password.
  • Schema migration v2→v3: The salt column required a new DB version. onUpgrade drops and recreates the users table (items preserved). Existing plain-text rows cannot be retroactively hashed — re-registration is the correct security posture.
  • Background threading: All DB operations run on a single-threaded ExecutorService. postValue() crosses the thread boundary safely. Executors are shut down in onCleared() and onDestroyView().

Course Outcomes Addressed

Outcome 4 (MessageDigest, SecureRandom, ExecutorService APIs), Outcome 5 (credential security, rainbow table mitigation, adversarial schema design).
