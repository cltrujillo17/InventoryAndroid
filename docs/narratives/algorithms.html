Artifact Description

The artifact selected for this enhancement is the same Android Inventory Management application used in Milestone Two, originally developed during CS 360: Mobile Architecture and Programming. It is written in Java, targets Android API 34, and uses SQLite via the SQLiteOpenHelper class for local data persistence. The Milestone Two enhancement refactored the application's architecture to follow the MVVM pattern and, crucially, connected the inventory screen to the database for the first time. That foundation made Milestone Three possible: once data was reliably persisted and retrieved, meaningful search, sort, and analysis algorithms could be added on top of it.

Justification, Skills, and Improvements

This artifact was selected for the Algorithms and Data Structures category because the original implementation contained no search, filter, or sort functionality whatsoever. The inventory was loaded as a flat, unordered list with no way to find a specific item or identify items that needed restocking.

The first algorithmic improvement was a real-time search filter using a SQL LIKE query. A TextWatcher on the search field calls InventoryViewModel.setSearch() on every keystroke, which triggers a parameterized LIKE clause with wildcard matching on the item title column. Because the title column is indexed, the LIKE lookup operates sub-linearly on the stored dataset. The use of a parameterized query also prevents SQL injection.

The second improvement was a sort capability via SQL ORDER BY. A Spinner exposes four sort modes: newest first, alphabetical by name, quantity low-to-high, and quantity high-to-low. Each option maps to a validated string constant defined in AppDbHelper, and the ORDER BY clause is assembled from a whitelist of known values — no user input is ever concatenated into SQL strings. Sorting is performed by the SQLite engine rather than by a Java-level Collections.sort call.

The third improvement is a low-stock detection algorithm in InventoryViewModel. After every database reload, countLowStockItems() performs a single O(n) linear pass over the already-loaded item list and counts items whose quantity is at or below a defined threshold (currently five units). The result drives a warning banner and red quantity labels per card. Because this algorithm runs on the list just loaded from the database, it adds zero additional query overhead. The threshold constant is defined once and referenced from all three components.

Course Outcomes

This enhancement directly addresses Course Outcome 3: designing and evaluating computing solutions using algorithmic principles while managing tradeoffs. The tradeoff between in-memory sorting and database-level ORDER BY was a deliberate design decision. Course Outcome 4 is addressed through practical implementation of data filtering, sorting, and scanning using Android's SQLite API, the TextWatcher interface, and the Spinner component. Course Outcome 5 is supported by the use of parameterized queries for the search filter (SQL injection prevention) and by controlled ORDER BY construction from a constant whitelist.

Reflection

The most valuable learning from this milestone was developing a clearer sense of which layer of an application should be responsible for which type of computation. Working through this enhancement made it concrete why in-memory filtering is inefficient: the database engine is specifically designed to evaluate WHERE and ORDER BY clauses on indexed data. This principle — that computation should happen as close to the data as possible — is one I will carry forward.

The primary challenge was maintaining consistency across multiple components that all needed to know about the same threshold value. The solution was to define the constant once in InventoryViewModel as a public static field and reference it from all other locations, rather than hardcoding the number in three separate places. A secondary challenge was ensuring search and sort state survived configuration changes — because the state is stored in the ViewModel rather than in the Fragment, it persists automatically across screen rotations.
